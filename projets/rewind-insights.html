Rewind Insights

Un outil d'audit web gratuit qui note la performance, le SEO et l'accessibilité de n'importe quel site en moins d'une minute, sans inscription.

Le projet

Rewind Insights répond à un besoin simple : donner à n'importe qui un diagnostic clair de son site web, sans jargon ni compte à créer. On saisit une URL, l'outil interroge les métriques officielles de Google et renvoie un score sur 100 réparti sur trois axes (vitesse, SEO, expérience utilisateur), puis envoie un rapport par email.

Derrière cette simplicité apparente, le projet touche à toute la chaîne : une API Next.js côté serveur, une intégration à un service tiers (Google PageSpeed), une base de données PostgreSQL via Supabase, et un envoi d'emails transactionnels. C'est mon projet le plus complet, celui qui m'a le plus appris sur la mise en production.

Le cœur : la route d'audit

Tout part d'une route API qui reçoit une requête POST. Voici l'extrait clé, celui qui interroge Google et transforme sa réponse en un score exploitable.

src/app/api/audit/route.ts
export async function POST(request: Request) {
  const { email, url } = await request.json();

  // Validation défensive avant tout appel réseau
  if (!url)   return NextResponse.json({ error: 'Champ "url" manquant' },   { status: 400 });
  if (!email) return NextResponse.json({ error: 'Champ "email" manquant' }, { status: 400 });

  // Appel à l'API PageSpeed Insights de Google (stratégie mobile)
  const psUrl =
    `https://www.googleapis.com/pagespeedonline/v5/runPagespeed` +
    `?url=${encodeURIComponent(url)}` +
    `&strategy=mobile&key=${process.env.PAGESPEED_API_KEY}` +
    `&category=PERFORMANCE&category=SEO&category=ACCESSIBILITY`;

  const psRes = await fetch(psUrl);
  if (!psRes.ok) {
    const raw = await psRes.text();
    return NextResponse.json(
      { error: 'PageSpeed erreur HTTP', http_status: psRes.status, response_body: raw.slice(0, 600) },
      { status: 500 }
    );
  }

  const psData = await psRes.json();
  const cats = psData?.lighthouseResult?.categories;

  // On mappe les 3 catégories Lighthouse en un score /100
  const scores = {
    speed: Math.round((cats.performance?.score   ?? 0) * 100),
    seo:   Math.round((cats.seo?.score           ?? 0) * 100),
    ux:    Math.round((cats.accessibility?.score ?? 0) * 100),
  };
  // ...
}
À retenir : Une réponse d'API externe n'est jamais garantie. Le ?. (chaînage optionnel) et le ?? 0 (coalescence des nuls) protègent chaque accès : si Google change son format ou renvoie une catégorie vide, le code ne plante pas, il retombe sur une valeur sûre.

Chaque cas d'échec renvoie un code HTTP juste (400 pour une saisie invalide, 500 pour une panne externe) avec un message lisible. Quand quelque chose casse, ce message me dit où regarder.

Ne jamais perdre un lead

Une fois les scores calculés, deux choses doivent arriver : enregistrer le contact en base et lui envoyer son rapport. J'ai fait un choix délibéré sur l'ordre et la gestion d'erreur.

src/app/api/audit/route.ts
// 1. On sauvegarde le lead AVANT d'envoyer l'email
const { data, error: sbError } = await supabase
  .from('leads')
  .insert([{ email, url_site: url, scores, statut: 'done' }])
  .select('id')
  .single();

if (sbError) console.error('[audit] Échec insertion Supabase :', sbError.message);

// 2. L'envoi d'email est NON BLOQUANT :
//    si le SMTP tombe, le lead est déjà sauvé, on log sans crasher
try {
  await sendAuditEmail({ email, url, scores });
} catch (e) {
  console.error('[audit] Échec envoi email :', e);
}

return NextResponse.json({ success: true, scores }, { status: 201 });
Le raisonnement : L'email peut échouer pour mille raisons hors de mon contrôle. En sauvegardant le contact d'abord, puis en isolant l'envoi dans un try/catch, un problème de messagerie ne fait jamais perdre une donnée client.

Ce que j'en retiens

Ce projet m'a fait passer du « ça tourne sur mon PC » au « ça tourne pour de vrais utilisateurs ». J'y ai appris à gérer des variables d'environnement, à découper le code en modules réutilisables (un client Supabase, un module d'envoi d'email), et surtout à écrire du code défensif : anticiper ce qui peut mal tourner plutôt que de croiser les doigts.
